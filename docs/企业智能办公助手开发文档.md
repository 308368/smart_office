## 一、项目概述

### 1.1 项目背景

随着企业数字化转型的深入，员工日常工作中面临大量重复性咨询（如制度查询、流程办理、问题解答等），传统的人工客服模式效率低下且成本高昂。同时，企业内部沉淀了大量文档资料（规章制度、操作手册、技术文档等），但缺乏有效的检索手段，导致知识利用率低。

本项目旨在构建一款**企业级AI智能办公助手**，基于RAG（检索增强生成）架构，结合大语言模型能力，为企业提供智能问答、文档检索、工单处理、流程辅助等功能，实现企业知识的智能化管理与应用。

### 1.2 项目目标

1. **知识智能化**：将企业散落的各类文档转化为可检索、可问答的智能知识库
2. **办事效率提升**：通过AI助手快速解答员工疑问，减少人工咨询成本
3. **流程自动化**：智能处理工单、请假、报销等常见流程，提升审批效率
4. **数据安全保障**：支持私有化部署，确保企业数据不出内网

### 1.3 核心特性

| 特性 | 描述 |
|------|------|
| 智能问答 | 基于企业知识库的AI问答，支持多轮对话 |
| 文档解析 | 支持PDF、Word、Excel、PPT、Markdown等格式 |
| 工单处理 | 智能工单分类、分配、转派 |
| 实时推送 | WebSocket实时消息通知 |
| 权限管理 | 精细化的RBAC权限控制 |
| 审计日志 | 完整操作记录，支持审计追溯 |

### 3.2 核心功能详细设计

#### 3.2.1 智能问答模块

**功能描述：**
基于RAG架构的企业知识库智能问答，支持多轮对话、上下文理解、答案溯源。

**核心流程：**
```
用户提问 → 向量化问题 → ES相似度检索 → 获取Top-K文档 → 构建Prompt →
大模型生成回答 → 返回答案+引用来源 → 存储对话历史
```

**关键特性：**
1. **多轮对话**：支持上下文关联追问
2. **答案引用**：展示参考的知识片段
3. **对话风格**：正式/轻松可选
4. **敏感词过滤**：AI输出安全审核
5. **访问限流**：防止恶意刷屏

**接口设计：**

| 接口 | 方法 | 描述 |
|------|------|------|
| /api/ai/chat | POST | 发送问题获取回答 |
| /api/ai/chat/stream | GET | 流式回答（SSE） |
| /api/ai/history | GET | 获取对话历史 |
| /api/ai/history/{id} | DELETE | 删除单条对话 |

#### 3.2.2 知识库管理模块

**功能描述：**
企业知识库的创建、维护、文档解析与向量化。

**核心流程：**
```
文档上传 → 文件类型识别 → 解析提取文本 →
文本分chunk → 向量化 → 存入ES →
更新知识库索引 → 关联元数据存储
        ↓
 WebSocket通知前端解析完成
```

**功能特点：**
- 异步解析：文档上传后进入RabbitMQ队列异步处理
- 实时通知：通过WebSocket向前端推送解析进度和结果
- 失败重试：解析失败可重新上传文档

**文档解析支持：**
| 格式 | 处理方式 | 状态 |
|------|----------|------|
| PDF | PDFBox / Apache PDFBox | ✅ |
| Word | Apache POI | ✅ |
| Excel | Apache POI | ✅ |
| PPT | Apache POI | ✅ |
| Markdown | 自研解析器 | ✅ |
| TXT | 直接读取 | ✅ |

**接口设计：**

| 接口 | 方法 | 描述 |
|------|------|------|
| /api/knowledge/ kb/create | POST | 创建知识库 |
| /api/knowledge/ kb/{id} | GET | 获取知识库详情 |
| /api/knowledge/ kb/{id}/doc/add | POST | 上传文档 |
| /api/knowledge/ kb/{id}/doc/list | GET | 文档列表 |
| /api/knowledge/ kb/{id}/doc/{docId} | DELETE | 删除文档 |
| /api/knowledge/ kb/{id}/rebuild | POST | 重建索引 |

#### 3.2.3 工单处理模块

**功能描述：**
企业工单的提交、流转、审批全流程管理，集成AI自动分类。

**工单类型：**
- IT支持（设备故障、软件问题）
- 人事行政（入职、离职、证明开具）
- 财务报销（差旅、招待、采购）
- 流程审批（请假、调休、加班）

**核心流程：**
```
提交工单 → AI自动分类 → 智能分配 →
负责人处理 → 审批流转 → 办结归档
```

**AI分类示例：**
```
用户输入：我想请3天年假，从下周一開始
AI识别：
  - 类型：请假申请
  - 子类型：年假
  - 提取信息：开始日期、时长3天
  - 推荐流程：直属领导审批 → 人事备案
```

**接口设计：**

| 接口 | 方法 | 描述 |
|------|------|------|
| /api/ticket/create | POST | 提交工单 |
| /api/ticket/list | GET | 工单列表 |
| /api/ticket/{id} | GET | 工单详情 |
| /api/ticket/{id}/approve | POST | 审批通过 |
| /api/ticket/{id}/reject | POST | 审批拒绝 |
| /api/ticket/{id}/transfer | POST | 转派工单 |
| /api/ticket/ai/classify | POST | AI工单分类 |

## 六、核心代码设计

### 6.1 RAG检索核心流程

```java
@Service
@RequiredArgsConstructor
public class RagService {

    private final ElasticsearchTemplate esTemplate;
    private final DocumentParser documentParser;
    private final EmbeddingModel embeddingModel;
    private final ChatModel chatModel;

    /**
     * RAG检索核心流程
     */
    public ChatResponse ragQuery(Long kbId, String question) {
        // 1. 向量化问题
        Embedding questionEmbedding = embeddingModel.embed(question).content();

        // 2. ES相似度检索
        SearchResponse<DocumentChunk> response = esTemplate.search(s -> s
                .index("kb_chunk_" + kbId)
                .query(q -> q
                        .knn(k -> k
                                .field("vector")
                                .vector(questionEmbedding.vector())
                                .k(5)
                        )
                )
                .source(source -> source.filter(f -> f.includes("content", "docId", "chunkIndex")))
        , DocumentChunk.class);

        // 3. 构建上下文
        List<String> contexts = response.getHits().getHits().stream()
                .map(hit -> hit.getSource().getContent())
                .toList();

        // 4. 构建Prompt
        String prompt = buildPrompt(contexts, question);

        // 5. 调用大模型
        Prompt promptObj = new Prompt(prompt);
        ChatResponse chatResponse = chatModel.chat(promptObj);

        // 6. 封装引用来源
        List<Reference> references = response.getHits().getHits().stream()
                .map(hit -> Reference.builder()
                        .docId(hit.getSource().getDocId())
                        .chunkContent(hit.getSource().getContent())
                        .score(hit.getScore())
                        .build())
                .toList();

        return ChatResponse.builder()
                .answer(chatResponse.getResult().getOutput().getText())
                .references(references)
                .build();
    }

    /**
     * 构建RAG提示词
     */
    private String buildPrompt(List<String> contexts, String question) {
        return """
                你是一个企业智能助手，请根据以下参考资料回答用户问题。
                如果无法从资料中找到答案，请如实告知。

                参考资料：
                %s

                用户问题：%s

                要求：
                1. 回答要简洁明了
                2. 如有需要可以列出步骤
                3. 如果资料不足，请说明"根据现有资料无法确定..."

                回答：
                """.formatted(String.join("\n---\n", contexts), question);
    }
}
```

### 6.2 分布式限流设计

```java
@Component
public class ChatRateLimiter {

    private final RedisTemplate<String, String> redisTemplate;

    /**
     * 限流检查
     * @param userId 用户ID
     * @param limit 限制次数
     * @param window 窗口时间(秒)
     * @return 是否允许
     */
    public boolean tryAcquire(Long userId, int limit, int window) {
        String key = "ratelimit:chat:" + userId;
        LocalDateTime now = LocalDateTime.now();
        String windowKey = now.format(DateTimeFormatter.ofPattern("yyyyMMddHHmm"));

        Long count = redisTemplate.opsForZSet()
                .zAdd(key + ":" + windowKey, System.currentTimeMillis(), UUID.randomUUID().toString());

        // 设置过期时间
        redisTemplate.expire(key + ":" + windowKey, window, TimeUnit.SECONDS);

        return count != null && count <= limit;
    }
}
```
